# 인덱스(index)란?

인덱스는 책의 색인과 유사한 개념으로, 데이터베이스에서 특정 데이터를 빠르게 찾기 위한 자료구조입니다. 인덱스가 없다면 데이터베이스는 테이블의 모든 행을 순차적으로 검색해야 하는데(풀 테이블 스캔), 이는 대용량 데이터에서 매우 비효율적입니다.

---

## 인덱스의 작동 원리

인덱스는 주로 B-Tree, B+Tree 등의 자료구조를 사용하여 구현됩니다. 이러한 구조는 데이터를 정렬된 상태로 유지하며, 로그 시간 복잡도(O(log n))로 검색이 가능합니다.


## B-Tree (Balanced Tree)

B-Tree는 데이터베이스 인덱스에서 가장 널리 사용되는 자료구조 중 하나입니다.

### B-Tree의 특징

1. **균형 트리 구조**: 모든 리프 노드가 같은 레벨에 위치하여 검색 시간이 일정합니다.
2. **다중 키 저장**: 각 노드는 여러 개의 키를 저장할 수 있습니다.
3. **자식 노드 수**: n개의 키를 가진 노드는 최대 n+1개의 자식 노드를 가질 수 있습니다.
4. **정렬 상태 유지**: 모든 키는 정렬된 상태로 유지됩니다.
5. **모든 노드에 데이터 저장**: 내부 노드와 리프 노드 모두 키와 데이터(또는 데이터 포인터)를 저장합니다.

### B-Tree의 구조

```
         [키3, 키7]
        /     |     \
   [키1,키2] [키4,키6] [키8,키9]
```

### B-Tree의 작동 방식

1. **검색 과정**:
   - 루트 노드에서 시작하여 키 값을 비교합니다.
   - 검색 값이 노드의 키보다 작으면 왼쪽 자식으로, 크면 오른쪽 자식으로 이동합니다.
   - 이 과정을 반복하여 검색 값을 찾거나 리프 노드에 도달할 때까지 진행합니다.

2. **삽입 과정**:
   - 적절한 리프 노드를 찾아 키를 삽입합니다.
   - 노드가 가득 차면 분할(split)이 발생하여 중간 키가 부모 노드로 이동합니다.
   - 필요시 루트 노드까지 분할이 전파될 수 있습니다.

3. **삭제 과정**:
   - 키를 찾아 삭제합니다.
   - 노드의 키 수가 최소 요구치 미만이 되면 재분배(redistribution)나 병합(merge)이 발생합니다.

## B+Tree

B+Tree는 B-Tree의 변형으로, 데이터베이스 시스템에서 더 널리 사용됩니다.

### B+Tree의 특징

1. **데이터 저장 위치**: 모든 데이터는 리프 노드에만 저장됩니다.
2. **내부 노드 역할**: 내부 노드는 인덱스 역할만 하며 실제 데이터를 가리키는 포인터는 가지지 않습니다.
3. **리프 노드 연결**: 모든 리프 노드는 링크드 리스트로 연결되어 있어 순차적 접근이 용이합니다.
4. **중복 키**: 내부 노드의 키는 리프 노드에도 중복해서 존재합니다.
5. **높은 팬아웃(fan-out)**: 내부 노드가 데이터를 저장하지 않아 더 많은 키를 저장할 수 있으므로 트리의 높이가 낮아집니다.

### B+Tree의 구조

```
         [키3, 키7]
        /     |     \
   [키1,키2] [키4,키6] [키8,키9]
     |         |         |
   [데이터]   [데이터]   [데이터]
     ↔         ↔         ↔    (리프 노드 간 연결)
```

### B+Tree의 작동 방식

1. **검색 과정**:
   - B-Tree와 유사하게 루트에서 시작하여 키 값을 비교하며 내려갑니다.
   - 실제 데이터는 리프 노드에서만 찾을 수 있습니다.
   - 범위 검색 시 첫 값을 찾은 후 리프 노드의 링크를 따라 순차적으로 접근합니다.

2. **삽입 과정**:
   - 적절한 리프 노드를 찾아 키와 데이터를 삽입합니다.
   - 노드가 가득 차면 분할이 발생하고, 중간 키의 복사본이 부모 노드로 이동합니다.
   - 분할된 리프 노드들은 링크로 연결됩니다.

3. **삭제 과정**:
   - 리프 노드에서 키와 데이터를 삭제합니다.
   - 노드의 키 수가 최소 요구치 미만이 되면 재분배나 병합이 발생합니다.
   - 내부 노드의 키가 변경될 수 있습니다.

## B-Tree와 B+Tree의 주요 차이점

| 특성 | B-Tree | B+Tree |
|------|--------|--------|
| 데이터 저장 | 모든 노드에 저장 | 리프 노드에만 저장 |
| 내부 노드 | 키와 데이터(또는 포인터) 저장 | 키만 저장 (인덱스 역할) |
| 리프 노드 연결 | 연결되지 않음 | 링크드 리스트로 연결됨 |
| 키 중복 | 없음 | 내부 노드의 키는 리프 노드에 중복 존재 |
| 노드당 키 수 | 상대적으로 적음 | 더 많은 키 저장 가능 |
| 트리 높이 | 상대적으로 높음 | 상대적으로 낮음 |
| 범위 검색 | 비효율적 | 매우 효율적 |
| 메모리 사용 | 상대적으로 적음 | 상대적으로 많음 |

## 데이터베이스에서의 활용

### B-Tree 활용 사례
- 유니크 인덱스나 프라이머리 키 구현
- 메모리 제약이 있는 환경
- 정확한 일치 검색이 주로 필요한 경우

### B+Tree 활용 사례
- 대부분의 RDBMS 시스템(MySQL, PostgreSQL, Oracle 등)의 기본 인덱스 구조
- 범위 검색이 자주 필요한 경우
- 순차적 데이터 접근이 필요한 경우
- 대용량 데이터베이스 시스템

## 성능 특성

### B-Tree
- 정확한 일치 검색에 최적화
- 단일 레코드 검색 시 약간 더 빠를 수 있음
- 트리 높이가 상대적으로 높아 디스크 I/O가 더 많을 수 있음

### B+Tree
- 범위 검색에 최적화
- 순차적 접근이 매우 효율적
- 더 낮은 트리 높이로 인해 디스크 I/O 감소
- 캐시 효율성이 더 높음

B+Tree는 이러한 장점들 때문에 대부분의 현대 데이터베이스 시스템에서 인덱스 구현에 선호되는 자료구조입니다.

## 인덱스 유형

## 1. B-트리 인덱스 (B-Tree Index)

- **정의**: 가장 일반적으로 사용되는 인덱스 구조로, 균형 잡힌 트리 구조를 사용
- **특징**:
  - 자식 노드가 2개 이상인 균형 트리 구조
  - 데이터의 정렬 순서 유지
  - 대부분의 RDBMS에서 기본 인덱스 구조로 사용
- **장점**:
  - 범위 검색에 매우 효율적
  - 정렬된 결과 반환 가능
  - 등호(=), 부등호(<, >, <=, >=) 연산 모두 지원
- **단점**:
  - 데이터 변경 시 트리 재구성 필요
  - 인덱스 크기가 클 수 있음
- **사용 사례**:
  - 대부분의 일반적인 쿼리 최적화
  - 범위 검색이 필요한 경우
  - 정렬이 필요한 쿼리

## 2. 클러스터드 인덱스 (Clustered Index)

- **정의**: 테이블의 물리적 저장 순서를 인덱스 키 값에 따라 정렬하는 인덱스
- **특징**:
  - 테이블당 하나만 존재 가능
  - 실제 데이터가 인덱스 키 순서대로 물리적으로 정렬됨
  - 주로 기본 키(Primary Key)에 자동 생성
  - B-트리 구조로 구현됨
- **장점**:
  - 범위 검색에 매우 효율적
  - 데이터 접근 속도가 빠름
  - 인덱스 자체에 데이터가 포함되어 있어 추가 조회 불필요
- **단점**:
  - 데이터 삽입/수정 시 물리적 재정렬 필요
  - 대량 업데이트 시 성능 저하 가능
- **사용 사례**:
  - 기본 키(Primary Key)
  - 자주 범위 검색을 수행하는 컬럼
  - 정렬이 자주 필요한 컬럼

## 3. 비클러스터드 인덱스 (Non-Clustered Index)

- **정의**: 데이터와 별도로 존재하는 인덱스 구조
- **특징**:
  - 테이블당 여러 개 생성 가능
  - 실제 데이터와 별도로 존재하며 포인터를 통해 데이터에 접근
  - 추가 저장 공간 필요
  - B-트리 구조로 구현되는 경우가 많음
- **장점**:
  - 다양한 검색 패턴 지원 가능
  - 데이터 물리적 구조에 영향 없음
  - 여러 인덱스를 통한 다양한 접근 경로 제공
- **단점**:
  - 클러스터드 인덱스보다 검색 속도 느림 (추가 조회 필요)
  - 인덱스 유지 관리 비용 발생
- **사용 사례**:
  - 자주 조회되지만 수정은 적은 컬럼
  - WHERE 절에 자주 사용되는 컬럼
  - 외래 키(Foreign Key) 컬럼

## 4. 복합 인덱스 (Composite Index)

- **정의**: 두 개 이상의 컬럼을 조합하여 생성한 인덱스
- **특징**:
  - 컬럼 순서가 매우 중요 (선행 컬럼부터 사용해야 효과적)
  - B-트리 구조로 구현됨
  - 클러스터드 또는 비클러스터드 형태로 생성 가능
- **장점**:
  - 여러 컬럼 조합 쿼리 최적화
  - 단일 인덱스보다 선택성(selectivity) 향상
  - 인덱스 수를 줄여 관리 효율성 증가
- **단점**:
  - 컬럼 순서에 따라 성능 차이 큼
  - 첫 번째 컬럼이 조건에 없으면 효율 감소
- **사용 사례**:
  - 여러 컬럼을 함께 조회하는 쿼리
  - `WHERE col1 = ? AND col2 = ?` 형태의 쿼리

## 5. 해시 인덱스 (Hash Index)

- **정의**: 해시 함수를 사용하여 키를 버킷에 매핑하는 인덱스
- **특징**:
  - 정확한 일치 검색에만 효과적
  - 범위 검색에는 부적합
  - 메모리 기반 데이터베이스에서 주로 사용
- **장점**:
  - 매우 빠른 키-값 조회 (O(1) 시간 복잡도)
  - 동등 비교에 최적화
- **단점**:
  - 범위 검색, 정렬, 부분 일치 검색 불가
  - 해시 충돌 관리 필요
- **사용 사례**:
  - 정확한 일치 검색이 대부분인 경우
  - 메모리 기반 데이터베이스
  - 캐시 시스템

## 6. 커버링 인덱스 (Covering Index)

- **정의**: 쿼리에 필요한 모든 컬럼이 인덱스에 포함된 경우
- **특징**:
  - 데이터 페이지 접근 없이 인덱스만으로 쿼리 처리 가능
  - 실제 인덱스 유형이 아닌 인덱스 사용 방식
  - 비클러스터드 인덱스에서 주로 활용
- **장점**:
  - 매우 빠른 쿼리 성능 (테이블 접근 불필요)
  - I/O 비용 대폭 감소
- **단점**:
  - 인덱스 크기 증가
  - 인덱스 유지 관리 비용 증가
- **사용 사례**:
  - 자주 실행되는 중요 쿼리 최적화
  - 집계 함수를 사용하는 쿼리

## 7. 전문 검색 인덱스 (Full-Text Index)

- **정의**: 텍스트 문서 내용을 검색하기 위한 특수 인덱스
- **특징**:
  - 역인덱스(Inverted Index) 구조 사용
  - 토큰화, 형태소 분석 등 텍스트 처리 포함
  - 대용량 텍스트 데이터 검색에 최적화
- **장점**:
  - 자연어 검색 지원
  - 관련성 점수 제공
  - 부분 일치, 유사 검색 가능
- **단점**:
  - 인덱스 크기가 큼
  - 업데이트 비용 높음
- **사용 사례**:
  - 블로그 검색
  - 문서 내용 검색
  - `WHERE MATCH(content) AGAINST('keyword')` 형태의 쿼리

## 실무에서의 인덱스 활용

### 대규모 서비스에서의 인덱스 전략

#### 1. 트랜잭션 처리 시스템 (OLTP)
- 자주 조회되는 컬럼에 선택적 인덱스 적용
- 복합 인덱스를 통한 다양한 쿼리 패턴 지원
- 인덱스 선택성(Selectivity) 고려: 카디널리티가 높은 컬럼 우선

```sql
-- 주문 조회가 자주 발생하는 경우
CREATE INDEX idx_orders_customer_date ON orders(customer_id, order_date);
```

#### 2. 분석 시스템 (OLAP)
- 데이터 웨어하우스에서의 컬럼 스토어 인덱스
- 파티셔닝과 결합한 인덱스 전략
- 집계 함수와 함께 사용되는 컬럼에 인덱스 적용

#### 3. 소셜 미디어 플랫폼
- 사용자 활동 타임라인 조회를 위한 시간 기반 인덱스
- 해시태그 검색을 위한 인버티드 인덱스
- 그래프 데이터베이스의 관계 인덱스

#### 4. 전자상거래 플랫폼
- 상품 검색을 위한 다차원 인덱스
- 가격 범위 검색을 위한 인덱스
- 사용자 구매 이력 조회를 위한 복합 인덱스

## 성능 측면에서의 인덱스

### 인덱스의 장점

1. **검색 속도 향상**: O(n) → O(log n)으로 시간 복잡도 감소
2. **정렬 비용 감소**: 이미 정렬된 인덱스 활용
3. **조인 성능 향상**: 조인 키에 인덱스가 있으면 해시 조인이나 중첩 루프 조인 성능 개선
4. **집계 함수 최적화**: MIN, MAX 등의 함수가 인덱스만으로 처리 가능

### 인덱스의 단점

1. **추가 저장 공간**: 인덱스는 추가 디스크 공간 필요 (원본 데이터의 10~20%)
2. **쓰기 성능 저하**: INSERT, UPDATE, DELETE 시 인덱스도 함께 업데이트 필요
3. **인덱스 관리 비용**: 단편화, 통계 업데이트 등 관리 필요

### 성능 최적화 전략

#### 1. 인덱스 선택 최적화
- 실행 계획 분석을 통한 인덱스 사용 패턴 파악
- 쿼리 패턴에 맞는 인덱스 설계
- 불필요한 인덱스 제거

#### 2. 인덱스 단편화 관리
```sql
-- MS SQL Server에서 인덱스 재구성
ALTER INDEX idx_name ON table_name REORGANIZE;
ALTER INDEX idx_name ON table_name REBUILD;
```

#### 3. 인덱스 힌트 활용
```sql
-- MySQL에서 인덱스 힌트 사용
SELECT * FROM users USE INDEX (idx_users_email) WHERE email = 'example@mail.com';
```

#### 4. 인덱스 커버링 쿼리 설계
```sql
-- 인덱스 커버링 쿼리 예시 (idx_users_name_email 인덱스가 있다고 가정)
SELECT name, email FROM users WHERE name = '홍길동';
```

## 대규모 서비스에서의 실제 사례

### 1. SNS 플랫폼 (예: 페이스북, 트위터)
- 타임라인 조회를 위한 사용자 ID + 시간 기반 복합 인덱스
- 해시태그 검색을 위한 역인덱스(Inverted Index)
- 그래프 탐색을 위한 관계 인덱스

### 2. 전자상거래 (예: 아마존, 쿠팡)
- 상품 카테고리 + 가격 + 평점 복합 인덱스
- 전문 검색(Full-text search)을 위한 인덱스
- 지리적 위치 기반 검색을 위한 공간 인덱스

### 3. 금융 시스템
- 트랜잭션 ID + 시간 기반 인덱스
- 계좌 번호 인덱스 (높은 선택성)
- 시계열 데이터를 위한 파티셔닝 + 인덱스 조합

### 4. 로그 분석 시스템
- 시간 범위 검색을 위한 인덱스
- 이벤트 타입 + 시간 복합 인덱스
- 샤딩된 환경에서의 분산 인덱스

## 인덱스 모니터링 및 유지보수

### 1. 인덱스 사용량 모니터링
```sql
-- MySQL에서 인덱스 사용 통계 확인
SHOW INDEX FROM table_name;
```

### 2. 미사용 인덱스 식별
```sql
-- PostgreSQL에서 미사용 인덱스 확인
SELECT * FROM pg_stat_user_indexes WHERE idx_scan = 0;
```

### 3. 인덱스 단편화 관리
- 주기적인 인덱스 재구성 또는 재빌드
- 자동화된 유지보수 작업 스케줄링
